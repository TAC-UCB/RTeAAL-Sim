# RTeAAL Sim Kernel

This directory contains the C++ simulation kernels and associated build infrastructure for RTeAAL Sim.

## Compiler Output Files

Running the RTeAAL Sim compiler (`make <design>`) on a FIRRTL file produces the following:

### `yaml/` — Fiber Tree Representation
YAML files encoding the sparse fiber tree representation of the RTL design.
These are the raw output of the Scala compiler and serve as an intermediate format:
- `operationMask_<design>_*.yaml` — fiber tree with ranks I (layer), S (layer output index), N (operation type)
- `operationInputMask_<design>_*.yaml` — fiber tree with ranks I (layer), S (layer output index), O (input operand order), R (input operand index)

### `txt/` — Layer Inputs and Mux Jump Tables
- `layerIn_<design>.txt` — initialized layer inputs for the simulation
- `muxJT_<design>.txt` — jump table for mux chain evaluation

### `kernel_IU_<design>.cc`, `kernel_SU_<design>.cc`, `kernel_TI_<design>.cc`
Design-specific kernel source files generated by the RTeAAL Sim compiler.
Unlike the generic kernels (RU, NU, OU, PSU), which produce a single binary shared across all RTL designs, these kernels are emitted per design by the Scala compiler and must be recompiled for each new RTL design.

### `logs/` — Timing and Memory Logs
- `tensor_<design>.log` — timing and peak memory usage of the Scala compiler generating tensor files from FIRRTL
- `compile_kernel_<variant>_<design>.log` — timing and peak memory usage of compiling each kernel variant

## Tensor Preprocessing

The raw YAML fiber tree files are not directly consumed by the kernels.
They must first be processed by `preprocess_tensor.cc` to produce JSON files in `json/`:

```bash
./bin/preprocess_tensor <design>
```

This is run automatically as part of `make <design>`.
The resulting JSON files (e.g., `*_sptr.json`, `*_rptr.json`, `*_nptr.json`) are the actual inputs read by the simulation kernels at runtime.

## Adding a New RTL Design

Follow these steps to add a new RTL design to RTeAAL Sim:

**1. Add the FIRRTL file**

Place the design's `.fir` file in `essent/firrtls/`.

**2. Register the design in the Makefile**

In `kernel/Makefile-gen.mk`:
- Add the design name to the `DESIGNS` list
- Add a `FIRRTL_<design>` variable pointing to its `.fir` file

Example:
```makefile
DESIGNS = \
    ... \
    mydesign-1c \

FIRRTL_mydesign_1c := $(firrtl_dir)/freechips.rocketchip.system.MyDesignConfig.fir
```

**3. Register the FIRRTL filename in the Scala compiler**

In `src/main/scala/Compiler.scala` at line 273, add a new `else if` branch mapping the FIRRTL filename to the design name:
```scala
} else if (designfileName == "freechips.rocketchip.system.MyDesignConfig.fir") {
  println("[INFO] My Design.")
  designName = "mydesign-1c"
}
```

**4. Recompile the RTeAAL Sim compiler**

```bash
cd kernel
make build_jar
```

Then run the compiler on the new design to generate tensor files and the design-specific kernel sources:
```bash
make mydesign-1c
```

This also generates a `TestHarness.h` header file for the design.

**5. Extract memory instance metadata**

In `utils/bin/`, run the header parser to extract memory instance information from `TestHarness.h`:
```bash
cd utils/bin
python3 header_parser.py <design>
```

**6. Update memory array sizes in `kernel_*.cc`**

Each kernel requires the sizes of the `uint8_t` and `uint64_t` memory arrays to be declared statically.
Look up the `size8` and `size64` values produced by the previous step and add a new branch in each `kernel/kernel_*.cc` file:
```cpp
} else if (prefix == "mydesign-1c") {
    size8  = <value from header_parser>;
    size64 = <value from header_parser>;
}
```
